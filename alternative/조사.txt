Cisco Packet Tracer 웹 구현 조사 및 설계 초안
작성 시각: 2026-06-17 KST

1. 요구사항 요약

- Cisco Packet Tracer처럼 사용자가 라우터, 스위치, 방화벽, PC, 서버, 무선 장비, 허브, 케이블을 배치하고 연결할 수 있는 웹 네트워크 실습 프로그램을 만든다.
- 각 장비는 클릭해서 설정할 수 있어야 한다.
- 라우터/스위치/방화벽 같은 네트워크 장비는 CLI 입력, 전원 끄기/켜기, 모듈 장착/제거를 지원해야 한다.
- 사용자별 저장 기능과 로그인/회원가입이 필요하다.
- 메인 페이지는 로그인한 사용자의 기존 프로젝트를 보여 주고, 없으면 새 네트워크 구성 시작 흐름을 제공해야 한다.
- 네트워크는 단순 그림이 아니라 실제로 동작해야 한다. 최소한 ARP, ICMP ping, 스위칭, 라우팅, DHCP, DNS, 서버 서비스 같은 학습용 흐름이 화면에서 검증되어야 한다.
- `.pkt` 파일 내보내기 요구가 있다. 다만 조사 결과 `.pkt` 완전 호환은 가장 큰 리스크다.

6. 시뮬레이션 엔진 설계

- 화면 토폴로지와 엔진 모델을 분리한다.
- 엔진은 다음 계층으로 나눈다.
  - Physical/link layer: 포트, 케이블 타입, link up/down, bandwidth, duplex, 전원 상태
  - L2: Ethernet frame, MAC address, ARP, switch MAC table, VLAN 기본 모델
  - L3: IPv4 packet, subnet match, default gateway, static route, router forwarding
  - L4/L7: DHCP, DNS, HTTP, ICMP 중심의 학습용 프로토콜
  - Simulation event queue: Event List와 PDU animation을 위한 시간순 이벤트 큐
- Realtime Mode:
  - 구성 변경 후 즉시 reachability를 재계산한다.
  - ping 같은 사용자 액션은 결과를 빠르게 보여 준다.
- Simulation Mode:
  - 사용자가 source/destination을 선택하면 PDU를 생성한다.
  - Capture/Forward는 이벤트 큐에서 한 단계씩 진행한다.
  - Auto Capture/Play는 일정 간격으로 단계 진행한다.
  - 각 이벤트는 time, lastDevice, atDevice, type, layer info, status를 가진다.

6-1. 로컬 모델 문서에서 확인한 프로토콜 구현 근거

- `model_layer2.htm`
  - 스위치는 수신 프레임의 access/trunk 포트 타입과 Dot1Q 포맷을 비교한다.
  - VLAN별 MAC table을 유지하고, source MAC이 없으면 학습하며 타이머 만료 시 제거한다.
  - destination MAC이 unicast이면 MAC table에 따라 전달하고, multicast/broadcast이면 같은 VLAN의 허용 포트로 flood한다.
  - blocking 포트는 STP 프레임 외 일반 프레임을 drop한다.
- `model_misc.htm`
  - 장비가 IP packet을 보낼 때 ARP table을 먼저 확인한다.
  - ARP entry가 없으면 원 IP packet을 drop/대기시키고 ARP request를 broadcast한다.
  - ARP request는 포트 down, IP 미설정, 같은 요청 진행 중이면 보내지 않는다.
  - NAT는 inside/outside 포트와 NAT table을 기준으로 TCP/UDP/ICMP 포트 및 주소 변환을 한다.
- `model_layer7.htm`
  - DHCP client는 DHCP-DISCOVER를 보내고, OFFER를 받으면 REQUEST를 보내며, ACK를 받으면 IP/subnet/gateway를 설정한다.
  - DHCP server는 DISCOVER에 OFFER, REQUEST에 ACK, RELEASE에 lease 해제를 수행한다.
  - DHCP 서버에 사용 가능한 IP가 없거나 잘못된 메시지는 drop한다.
- `model_layer3Routing.htm`
  - RIP은 포트 up, RIP enabled, passive interface가 아닌 조건에서 update/request를 처리한다.
  - RIPv1/RIPv2는 broadcast/multicast와 수신 가능 버전이 다르게 동작한다.
  - EIGRP/OSPF는 process enabled, 같은 subnet, passive interface 여부, neighbor 상태 등을 검사한다.
- 1차 구현은 위 흐름 중 CCNA 실습에 자주 쓰는 ARP, ICMP, MAC learning, VLAN access/trunk, static route, DHCP를 먼저 구현한다.
- RIP/OSPF/EIGRP는 UI와 CLI 명령은 받을 수 있게 두되, 실제 동적 라우팅 수렴은 2차 기능으로 둔다.

6-2. 1차/2차 기능 경계

- 1차에서 반드시 실제 동작시킬 기능:

  - Hub flood와 단순 collision 표시

  - IPv4 address/subnet/default gateway 설정
  - ARP request/reply와 ARP table 표시
  - ICMP ping 성공/실패
  - Static route 기반 라우터 forwarding
  - DHCP discover/offer/request/ack와 lease table
  - DNS A record 조회
  - HTTP service reachability 확인

- 1차에서 UI/설정만 만들고 실제 엔진은 단순화할 기능:
  - Wireless association: SSID/WEP 일치 여부 중심, 실제 전파 물리 계산 없음

  - NAT inside/outside 실제 translation table
  - TCP 상태 머신, FTP/Email/TFTP/SNMP/VPN/VoIP

8. `.pkt` 호환 조사와 결론

- 샘플 `.pkt` 경로:
  - `C:\Program Files (x86)\Cisco Packet Tracer 6.1sv\saves\TCP_IP\TCP_IP.pkt`
- 조사 결과:
  - `file` 명령은 단순 `data`로 식별했다.
  - `xxd` 첫 바이트에 JSON/XML/ZIP 매직이 없다.
  - `strings`에서 의미 있는 평문 구조가 거의 나오지 않았다.
  - 설치본의 `unzip.exe`로 열었을 때 ZIP 중앙 디렉터리 시그니처가 없다는 오류가 나왔다.
- 결론:
  - `.pkt`는 공개 표준 파일이 아니며, 완전 호환 exporter를 바로 구현하는 것은 현실적으로 매우 위험하다.
  - 웹앱 내부 저장 포맷은 JSON 기반으로 안정적으로 설계한다.
  - `.pkt`처럼 보이게 저장하면 Packet Tracer 6.1에서 "not compatible" 오류를 만들 수 있으므로, 1차 구현에서는 자체 메타데이터/토폴로지 데이터를 `.ptweb` 확장자로 저장한다.
  - 완전 호환이 필요하면 Cisco가 공개한 포맷 문서나 공식 SDK가 필요하다. 현재 조사 범위에서는 그런 문서를 확인하지 못했다.
  - 따라서 `.pkt` 관련 코드는 `exporters/packetTracer` 같은 독립 모듈로 격리하고, 추후 포맷 정보가 생기면 교체한다.

8-1. IPC/Script Module 조사

- 설치 폴더의 `extensions`에는 IPC/Script 관련 파일이 있다.
  - `extensions/IpcTest/IpcTest.xml`
  - `extensions/IpcTest/IpcTest.pta`
  - `extensions/IpcLog/IpcLog.pta`
  - `extensions/scriptTemplates/*.ptst`
- `IpcTest.xml`은 External Application 메타데이터이며, GET_NETWORK_INFO, CHANGE_NETWORK_INFO, SIMULATION_MODE, FILE, IPC 같은 privilege를 선언한다.
- 로컬 help의 `scriptModules_scriptEngine.htm`은 Packet Tracer 엔진/GUI가 IPC API로 접근 가능하고, API 선언이 `.pki` 파일에 있다고 설명한다.
- 하지만 설치 폴더 전체에서 `.pki` 파일은 발견되지 않았다.
- `.pta`, `.ptst` 파일은 사람이 읽을 수 있는 XML/JSON이 아니라 `data`로 식별되고 문자열도 의미 있게 나오지 않았다.
- 따라서 현 단계에서 Packet Tracer IPC를 이용해 웹 토폴로지를 자동으로 `.pkt`로 생성하는 방법은 확인되지 않았다.

9. 개발 스택 결정

- `alternative/개발언어추천.md`의 이상적 권장:
  - Frontend: TypeScript + React
  - Simulation Engine: Rust + WebAssembly
  - Backend: Go
  - DB: PostgreSQL
- 현재 로컬 환경:
  - Node.js 24.15.0
  - npm 11.12.1
  - Rust, Go, wasm-pack은 현재 확인되지 않음
- npm registry 확인 버전:
  - React 19.2.7
  - Vite 8.0.16
  - @vitejs/plugin-react 6.0.2
  - lucide-react 1.20.0
- 1차 완성 스택:
  - TypeScript + React + Vite
  - 시뮬레이션 엔진도 TypeScript로 작성하되, `engine` 디렉터리 경계를 분리해 나중에 Rust/WASM으로 교체 가능하게 한다.
  - 사용자/프로젝트 저장은 localStorage 기반으로 먼저 완성한다.
  - UI와 엔진 데이터 모델은 순수 JSON으로 연결한다.
- 이유:
  - 현재 빈 프로젝트에서 가장 빨리 실제로 작동하는 웹앱을 만들 수 있다.
  - 설치 가능한 도구가 Node/npm뿐이라 빌드/검증 실패 위험이 낮다.
  - 요구사항 대부분은 웹 UI와 상태 관리, 이벤트 시뮬레이션으로 먼저 구현 가능하다.

10-1. Simulation UX 세부

- Packet Tracer는 User Created Packet Window(UCPW)와 Event List를 구분한다.
- UCPW:
  - 사용자가 만든 PDU를 저장한다.
  - scenario별로 source, destination, type, color, time, periodic, last status를 가진다.
  - reset simulation을 눌러도 사용자 PDU 목록은 지워지지 않는다.
- Event List:
  - 네트워크에서 실제로 발생한 모든 PDU 이벤트를 기록한다.
  - 사용자가 만든 PDU뿐 아니라 ARP, DHCP, RIP, OSPF 같은 프로토콜 자동 이벤트도 표시한다.
- PDU detail:
  - OSI Model
  - Inbound PDU Details
  - Outbound PDU Details
  - 각 layer의 처리 결과: encapsulate, de-encapsulate, transfer, accept, queue, drop, transmit
- 구현 방향:
  - `simulation.scenarios`에는 사용자 PDU와 예약 시간을 저장한다.
  - `simulation.events`에는 실행 결과 이벤트를 저장한다.
  - Event 클릭 시 OSI layer 처리 로그와 header-like details를 보여 준다.
